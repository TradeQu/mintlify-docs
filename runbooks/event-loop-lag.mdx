## What fired

The API process's Node.js event-loop lag crossed its threshold.

| Alert | Condition | For |
| --- | --- | --- |
| `MeshQuEventLoopLag` (warning) | event-loop lag > **100ms** | 5 min |

Event-loop lag means the runtime cannot process work promptly — synchronous CPU-bound work or resource starvation is blocking the loop. It surfaces to clients as latency.

## Symptoms

- Elevated request latency across many routes — cross-check [high-latency](/runbooks/high-latency).
- Event-loop-lag panel above the threshold line.
- Often paired with [high-memory](/runbooks/high-memory) under resource pressure.

## First actions

1. **Check for cascade.** Lag, latency, and memory usually move together. Look at the cluster of signals, not just this one.
2. **Correlate with traffic and deploys.** A traffic burst can saturate a single instance; a recent deploy that introduces a heavy synchronous code path can block the loop. Roll back if onset lines up.
3. **Scope to instance vs fleet.** Is one instance lagging or all of them? A single bad instance can often be cycled out.
4. **Mitigate.** Cycling the affected instance clears transient lag as a stopgap. Capture state first; a persistent pattern needs root-cause investigation, not repeated restarts.
