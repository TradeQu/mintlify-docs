## What fired

The MeshQu API's server-error ratio crossed an SLO threshold.

| Alert | Condition | For |
| --- | --- | --- |
| `MeshQuHighErrorRate` (critical) | 5xx responses > **1%** of all requests | 5 min |
| `MeshQuElevatedErrorRate` (warning) | 5xx responses > **0.5%** of all requests | 10 min |

The error-rate SLO target is **< 0.1%** over a 5-minute window. Sustained 5xx above SLO is treated as a service-degradation incident.

## Symptoms

- Clients see `500`/`502`/`503` responses from `api.meshqu.com`.
- Evaluate / record calls intermittently fail or time out.
- Error-rate panels on the MeshQu API dashboard are above the threshold line.

## First actions

1. **Confirm it's real.** A deploy mid-flight or a single bad instance can spike the ratio briefly. Check whether the rate is sustained or already recovering.
2. **Scope it.** Is it one route, one status code, or system-wide? Narrow by route and by status to find the blast radius.
3. **Check recent changes.** Correlate the onset with the most recent deploy or config change. A recent rollout is the most common cause — roll back if it lines up.
4. **Check dependencies.** Database, cache, and downstream availability. A failing dependency surfaces as 5xx at the API edge.
5. **Decide severity.** Sustained 5xx above SLO is an incident. If unsure, round up — it is easier to de-escalate than re-escalate.

## What this is not

MeshQu is **advisory by design** — it returns verdicts, it does not block or allow traffic in your application. A high error rate on the MeshQu API affects your ability to *obtain* governed verdicts and receipts; it does not change enforcement decisions already made in your own system.
