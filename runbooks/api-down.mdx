## What fired

A MeshQu API instance has been unreachable to the monitoring system.

| Alert | Condition | For |
| --- | --- | --- |
| `MeshQuDown` (critical) | the scrape target reports the instance as down | 1 min |

This is a **critical** availability alert. A full outage that persists is an incident.

## Symptoms

- Clients cannot reach `api.meshqu.com` (connection failures, gateway errors).
- The availability / `up` panel shows the target down.
- Evaluate and record calls fail entirely.

## First actions

1. **Confirm scope.** One instance down vs the whole service. A single unhealthy instance behind a load balancer is very different from a total outage.
2. **Confirm it is the service, not the monitor.** A monitoring or scrape-path problem can fire `MeshQuDown` while the API is actually serving. Verify independently before declaring an outage.
3. **Check the platform.** Deploy state, health checks, and the hosting platform's status. A failed deploy or a platform incident is the common cause.
4. **Declare an incident** if a real outage is confirmed and persisting — open the incident channel and assign roles per the internal incident-response flow.
5. **Communicate.** A confirmed outage warrants a status-page update for integrators.
