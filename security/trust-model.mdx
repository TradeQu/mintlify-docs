MeshQu's trust model has four levels. Each level answers a stronger verification question. The current deployment operates at **Level 2**.

| Level | Question it answers | Status |
|-------|-------------------|--------|
| 0 | Has this record been changed? | Active |
| 1 | Who issued it? | Active |
| 2 | Can a third party confirm it existed? | **Current** |
| 3 | Was the signing key physically protected? | Planned |

---

## Level 0 — Deterministic Integrity

Every decision receipt includes a SHA-256 hash computed from the decision, its violations, the input context, and the frozen policy snapshot that was used. The hash is computed over canonical JSON (keys sorted recursively), so the same inputs always produce the same hash regardless of property order or serialisation quirks.

This means you can replay any decision later — feed it the same context and the same snapshot — and check that the hash matches. If someone changed the verdict, added a violation, or swapped the policy version, the hash won't match.

The replay endpoint is `POST /v1/decisions/{id}/replay`.

**What this proves:** the decision record has not been altered since it was created.

---

## Level 1 — Cryptographic Signatures

Hashing detects tampering, but still assumes trust in the system that produced the hash. Level 1 reduces that dependency.

Every receipt is signed with an Ed25519 private key. The signature covers the integrity hash, which already covers the full receipt context. Verification only requires the public key — no API call, no shared secret, and no network dependency.

Keys are versioned. Each receipt includes a `signature_kid` (for example, `msk_v1`) identifying the signing key. During rotation, both current and previous keys remain valid for verification. See [Key Management](#key-management) below.

**What this proves:** the decision was issued by MeshQu and has not been modified by anyone since.

---

## Level 2 — Transparency Anchoring

<Tip>This is the **current** deployment level.</Tip>

Levels 0 and 1 prove a receipt has not been tampered with, but they still depend on MeshQu-operated infrastructure.

Level 2 anchors each signed receipt to [Sigstore Rekor](https://rekor.sigstore.dev), a public append-only transparency log. Rekor uses Merkle-tree transparency mechanisms similar to Certificate Transparency.

Once a receipt is anchored, anyone can query Rekor by entry UUID and verify the inclusion proof using public data.

Anchoring happens after signing and is not included in the integrity hash. If Rekor is unreachable, the receipt is still signed and valid — it just won't have a transparency anchor. Signing is never blocked by log availability.

<Note>
**Why DSSE envelopes?** Ed25519 is incompatible with Rekor's `hashedrekord` entry type ([sigstore/rekor#851](https://github.com/sigstore/rekor/issues/851)). MeshQu uses DSSE envelopes wrapping in-toto attestation statements instead.
</Note>

<Warning>
**Browser verifier scope:** MeshQu Verify validates entry existence, Merkle inclusion proofs, and Rekor's Signed Entry Timestamp (SET). It does not yet pin trust roots or verify checkpoint consistency over time in-browser.
</Warning>

**What this proves:** the decision existed at a specific point in time, recorded in a public log that MeshQu cannot retroactively alter.

## Actor Attribution Trust

When a decision is recorded with an `actor`, the following trust layers apply:

| Layer | Field | Basis | Trust level |
| --- | --- | --- | --- |
| Attestation | `actor` JSONB | Client-supplied claim | Application trust |
| Verified identity | `api_key_id` / `api_key_name` | Authenticated API key | Cryptographic proof |

The `actor.id` field is included in the integrity hash and is therefore **cryptographically bound** to the receipt. This means:

- If someone alters `actor.id` after the fact, the integrity hash will not match.
- Other actor fields (`type`, `role`, `authority`, `display_name`) are stored and displayed for audit convenience but are **not** part of the cryptographic proof — they can be viewed as application-level annotations.

<Warning>
MeshQu does not authenticate `actor.id` against any identity provider. It records and cryptographically binds whatever value you supply. Your application is responsible for ensuring the actor identity is accurate at call time.
</Warning>

The authenticated API key (`api_key_id`) identifies the **calling system**. The `actor` field identifies the **responsible party** within that system. Both are captured on every recorded decision.

## Level 3 — Hardware-Bound Identity

<Warning>Status: **Planned**.</Warning>

Levels 0–2 assume software-backed signing keys. Level 3 moves signing into tamper-resistant hardware (HSM or cloud KMS) so keys can sign without being exported.

This enables stronger identity binding and a path toward higher-assurance compliance profiles.

**What this proves:** cryptographic operations were performed inside tamper-resistant hardware. The signing key has not been extracted or cloned.

## Key Management

Every receipt includes a `signature_kid` identifying which key signed it.

MeshQu publishes versioned public signing keys so receipts remain verifiable across key rotation.

**Rotation:**

- During rotation, both current and previous keys remain valid for verification
- Old public keys are retained indefinitely — receipts never become unverifiable due to rotation

**Compromise response:**

If a signing key is compromised, MeshQu revokes that key and publishes updated verification material. Existing receipts can be re-verified via the [replay endpoint](/api/reference).

| Property | Value |
|----------|-------|
| Algorithm | Ed25519 (RFC 8032) |
| Key format | SPKI DER, base64-encoded |
| Key ID scheme | `msk_v<N>` (e.g. `msk_v1`) |
| Resolution | Versioned public key distribution |
